Computing Results¶

Overview¶

There are three options for computing the simulation results depending on the stage of the analysis. All options can be accessed from the Simulate menu or by right-clicking on a simulation in a workspace on the Simulation Manager.

  • Simulation Pre-check: The simulation pre-check is a way to quickly check an input file for warnings and errors, missing and out-of-range inputs, unusual material properties, and mesh quality issues.
  • Preprocessing: The preprocessor gives the user access to detailed plots showing the initial conditions.
  • Running the Solver: The solver computes the solution to the specified governing equations for the specified properties, geometry, initial conditions, and boundary conditions.

Jobs (a request to preprocess or run a simulation) are submitted to one of several possible queues (Preprocessing, Simulations, or Postprocessing) on the local machine and each connected machine (see Remote Solving Setup). The jobs in each queue are then run sequentially according to the user-specified maximum number of simultaneous simulations and the available license tokens. Each queue operates independently, so multiple jobs can be running concurrently in different queues, as shown below.

The buttons below the queue allow the user to pause, resume, terminate, and re-prioritize jobs in the queue. The controls are described below:

ICON ACTION
Pause the selected simulation. Either the pause or play icon will be shown at a time depending on the state of the simulation (paused or simulating).
Resume the selected simulation. Either the pause or play icon will be shown at a time depending on the state of the simulation (paused or simulating).
Terminate the selected simulation (if running) or remove the selected simulation from the queue (if queued but not running).
Move the selected simulation higher in the queue.
Move the selected simulation lower in the queue.
Download the results from the remote server. This is only active after the simulation completes or is terminated.

Options to Run Batch Process, Generate Batch Report, and Run Simulation and Batch Process are also available from the Simulate and right-click menus. From the right-click menu, the user can also access Simulation completion details. By the selecting this option, the user can access information for the Completion date and Completion time.

The current status of any registered remote servers (see Remote Solving Setup) can be accessed from the Simulate‣Show Server Loads… menu. This will show the current number of simulations running on the remote server, the available CPU, and the available memory as shown below.

Simulation Pre-check¶

The simulation pre-check is a way to quickly check an input file for Tips, Warnings, and Fatal Errors; Missing and Out-of-range Inputs; unusual Material Properties; and Mesh Quality. The idea is to run the simulation pre-check after the simulation setup is complete as a final check before submitting the simulation to the queue. Tips, Warnings, and Fatal Errors are reported on the General Model Settings tab, Missing and Out-of-range Inputs and unusual Material Properties are reported on the Active Material Properties tab of the Simulation Pre-check dialog, and the results of the Mesh Quality tests are reported on the Mesh Quality tab of the Simulation Pre-check dialog.

The simulation pre-check can be run by opening the Simulate menu, then choosing Simulation Pre-check. Alternatively, the simulation pre-check can be accessed on the Simulation Manager tab by right-clicking on a simulation in a workspace and choosing Simulation Pre-check. The results of the Simulation Pre-Check are summarized using color-coded flags: red flags indicate major (fatal) errors, yellow flags are cautionary, and green flags mean that no issues were detected.

Note

It should be kept in mind that the simulation pre-check is not a replacement for careful simulation setup - it is simply a tool to help identify potential issues early in the analysis process.

Tips, Warnings, and Fatal Errors¶

The Simulation Pre-Check runs the preprocessor tests to identify any settings that may be problematic. It reports the results on the General Model Settings tab as:

  • Tips: Tips are informational messages designed to give guidance about potential issues. These include warnings about units, the temperature where a linear thermal expansion coefficient will make the fluid density negative, etc.
  • Warnings: Warnings are cautionary messages that are more severe than the Tips described above. These indicate possible setup errors and accuracy issues. For example, messages about out-of-tolerance inputs (see Missing and Out-of-range Inputs) and mesh quality issues (see Mesh Quality) are reported here.
  • Fatal Errors: Fatal errors are major issues that will result in the preprocessor aborting. These include specifying erroneous values (e.g., a negative viscosity), missing key parameters (e.g. the fluid density), etc. Fatal errors will result in a red flag icon being shown.

Missing and Out-of-range Inputs¶

This part of the simulation pre-check feature tests for undefined variables, fractional inputs that are out-of-range and for correct variable relationships. Problematic variables are colored to quickly identify possible issues. Holding the mouse over a colored variable name or field will reveal a tool tip that gives more information about why the variable or field was flagged.

Warnings:

  • Fractional variables (e.g., the critical solid fraction) that are not between zero and one are highlighted in orange. If this is not corrected the preprocessor will reset these variables to the nearest in-range value (usually zero or one). The tool tip will report the valid range for the variable.
  • If the coherent solid fraction is greater than the critical solid fraction minus 0.1, , then both variable names and the fields are highlighted in orange. If this is not corrected the preprocessor will reset the coherent solid fraction to the nearest in-range value. The tool tip will report the valid range for the variable.

Critical issues:

  • The names of any required input variables that are undefined are displayed in red. The tool tip will describe these variables as Not set.
  • If the solidus temperature is greater than the liquidus temperature then both variable names and the fields are highlighted in red. If this is not corrected the preprocessor will disable the solidification model. The tool tip will report the valid range for the variable.

Note

Input variables are tested for validity before comparative tests are done. The comparative test is only done if the input is valid, otherwise the simulation pre-check will report why the input is invalid.

Material Properties¶

Material properties are compared to reference materials that are selected from the materials database. This allows for a quick comparison of the material properties to identify any that are considerably different from a known material, potentially identifying typographical errors or mismatched units. The names of the constant variables that are more than a specified percentage away from the same value for a reference material are highlighted in blue; the tool tip reports these as Out of tolerance. The default tolerance is 5%, but it can be changed by modifying the value in the Tolerance +/- box located in the upper-right corner of the Simulation Pre-Check dialog. The tolerance and the selected reference materials apply to all simulations and will be saved when the Save or Save & Close buttons are clicked.

The material properties can also be compared graphically: constant properties are compared to the reference materials using a bar graph, while tabular properties are shown using a scatter plot. Note that the Out of tolerance coloring is only applied to constant material properties, so a graphical comparison is the only way to assess whether tabular properties are reasonable. Some examples of the comparison plots are shown below. The order of the entries on the bar graph match the order of the reference materials. Additionally, the plot color for each material can be changed by left-clicking on the corresponding entry in the legend.


Mesh Quality¶

The simulation pre-check also performs some tests to determine the mesh quality. The results of the mesh quality tests are reported on the Mesh Quality tab of the Simulation Pre-check dialog. The Mesh Quality‣Mesh Block Properties tab shows issues within a single mesh block, whereas the Mesh Quality‣Inter-Block Data tab shows the issues associated with the interface between the current mesh block and the adjacent mesh block.

The quality of the mesh is assessed by comparing certain mesh parameters to the general guidelines defined below. These guidelines are not rigid requirements for a successful simulation, just recommended values that usually give good results for a wide variety of applications.

  • The maximum cell aspect ratio (e.g. , etc.) should be less than 3. When this guideline is exceeded, the mesh block name is highlighted in red and the reported Maximum aspect ratio for the direction that exceeded the limit is highlighted in red on the Mesh Quality‣Mesh Block Properties tab.
  • The maximum adjacent cell size ratio (e.g. , etc.) should be less than 1.25. When this guideline is exceeded, the mesh block name is highlighted in red and the reported Maximum adjacent cell size ratio for that mesh block is highlighted in red on the Mesh Quality‣Mesh Block Properties tab. The locations of the flagged cells can be viewed by clicking on the icon.
  • The maximum inter-block cell size ratio (e.g. , , etc.) should be less than 2. When this guideline is exceeded, the names of both of the mesh blocks are highlighted in red and the offending cell in the selected mesh block is listed in the Cell size ratio area on the Mesh Quality‣Inter-Block Data tab.

Figure 1: Cell dimensions used to define the cell aspect ratio, the adjacent cell size ratio, and the inter-block cell size ratio. Note that these dimensions are illustrative and that the simulation pre-check computes these in all three coordinate directions.

In addition to the above checks, the Mesh Quality check also identifies mesh planes that do not coincide between mesh blocks; i.e. Missing mesh planes are mesh planes that are not in the current mesh block but are in the adjacent mesh block. An example of a missing mesh plane is shown below; in this case, the red mesh block (mesh block 1) is missing a mesh plane in the z-direction from the blue mesh block (mesh block 2).

Once the simulation pre-check identifies a missing mesh plane, it will display the coordinate associated with the missing mesh plane and give an option to add this mesh plane to the current mesh block (mesh block 1, in the example case).

Preprocessing¶

The preprocessor gives the user access to detailed plots showing the initial conditions and converts the information in the input file into a format that the solver can use. It is very similar to the FAVORize option (), but it shows more information including:

  • Scalar variables: temperatures, pressures, densities, concentrations, etc.
  • Vector variables: velocities, stresses, etc.
  • Mesh information: FAVORized geometry, mesh resolution, conformed mesh blocks, etc.
  • Component properties: contact angle, roughness, velocities, etc.

Additionally, it creates several text diagnostic files that contain information about warnings, errors, and other parameters. These files and their content are discussed more in the Preprocessor Diagnostic Files section of the Troubleshooting chapter. Finally, the preprocessor checks the input file for incompatible options.

The preprocessor can be run by opening the Simulate menu, then choosing Preprocess Simulation, and then the desired machine. For example, the selection Simulate‣Preprocess Simulation‣Local would run the preprocessor on the local machine. Alternatively, the preprocessor can be accessed on the Simulation Manager tab by right-clicking on a simulation in a workspace, choosing Preprocess Simulation, and then the desired machine.

Note

The solver always calls preprocessor before running so it is not necessary to run the preprocessor separately before running the solver.

Running the Solver¶

The solver computes the solution to the selected governing equations for the specified properties, geometry, initial conditions, and boundary conditions. It generates several output files:

  • flsgrf.*: This is the main output file. It contains all of the spatial, history, and diagnostic data that was specified on the Model Setup‣Output tab.
  • flsplt.*: This is a diagnostic file that can be postprocessed graphically. It contains information about how the solver was running during the simulation.
  • hd3msg.*, hd3out.*, and hd3err.*: These are text files that contain information about solver warnings, errors, and other diagnostic parameters. These files and their content are discussed more in the Solver Diagnostic Files section of the Troubleshooting chapter.

Running a Simulation¶

Just like the preprocessor, the solver can be run by opening the Simulate menu, then choosing Run Simulation, and then the desired machine. For example, the selection Simulate‣Run Simulation‣Local would run the solver on the local machine. Alternatively, the solver can be accessed on the Simulation Manager tab by right-clicking on a simulation in a workspace, choosing Run Simulation, and then the desired machine.

It is also possible to run all of the simulations in a workspace sequentially by using the Simulate Workspace option. This will place all of the simulations that are in the workspace into the specified queue to be run. These simulations will then be run in sequence (in accordance with the specified Maximum number of simultaneous simulations) as license tokens become available. The Simulate Workspace option can be accessed through the Simulate menu or by right-clicking on a workspace on the Simulation Manager tab.

See also

  • See the Preprocessing and running simulations section for a guided tutorial on submitting jobs to the queue
  • See the Queue section for more detail on queuing simulations

Monitoring Simulation Progress¶

Diagnostic information indicating the status of the currently-selected simulation is displayed on the Simulation Manager tab in the Run-time plots and Solver Messages Window. It is described in more detail in the corresponding sections.

Note

The diagnostic information is made available while the simulation is running and is retained until the results files are deleted or overwritten.

Making Runtime Changes¶

It is sometimes helpful to change some solver settings while a simulation is running to make the solver run faster. This can be done using the Runtime options button on the Simulation Manager. Most, but not all, of the numerical options can be changed while the simulation is running. However, the data output rates or other parameters relating to the simulation setup are fixed and cannot be changed while the simulation is running.

There are several high-level controls on the Runtime changes to the solver dialog:

  • Send To Solver: This applies the selected changes to the simulation. A message will be printed in the hd3msg.* file when the solver reads and applies the requested changes.
  • Load the initial options: Populates the Runtime changes to the solver dialog with the options that were used when the simulation started.
  • Load last sent options: Populates the Runtime changes to the solver dialog with the options that were last applied to the simulation.
  • Save Options and Close: Saves the options on the Runtime changes to the solver dialog but does not apply them to the simulation.
  • Copy to simulation project file: Copies any changes made on the Runtime changes to the solver dialog to the prepin.* file.

To make the run-time changes:

  1. Select the simulation that you would like to modify in the queue.
  2. Press the Runtime Options button on the Simulation Manager tab to open the Runtime changes to the solver dialog.
  3. Make the desired changes to the solver parameters.
  4. Press the Send To Solver button to apply the changes.

Command Line Operations¶

This section will introduce the mechanics of running FLOW-3D in the command line mode. It covers using runscripts and displaying results, and describes how to use the solver monitoring utility PEEK. It also describes how to run a series of simulations using a single command. The symbol $ in this section represents a command prompt.

Runscripts¶

Runscripts (batch files) are provided to run the preprocessor, the solver, and the postprocessor. There is also a special runscript to run the preprocessor, solver, and postprocessor consecutively. Finally, another runscript can be used to run a series of simulations in sequence. The following runscripts are provided for running FLOW-3D :

  • runpre <ext1> <ext2>: Runs preprocessor on simulation prepin.<ext1> with the output files appended with the extension <ext2>. If <ext2> is not defined, then <ext1> is used for the output file extension. If neither <ext1> nor <ext2> are defined, the input simulation name is expected to be prepin.inp with the output files appended with <dat>.
  • runhyd <ext2>: Runs solver using <ext2> as the extension for both input (created by runpre) and output files. If <ext2> is not defined, then the input files are expected to have extension <dat> which is also used for the output files.
  • runpost <ext2>: Runs postprocessor using <ext2> as the extension for both input (created by runpre and runhyd) and output files. If <ext2> is not defined, then the input files are expected to have extension <dat> which is also used for the output files.
  • runall <ext1> <ext2>: Runs all three scripts on simulation prepin.<ext1> with the output files appended with the extension <ext2>. If <ext2> is not defined, then <ext1> is used for the output file extension. If neither <ext1> nor <ext2> are defined, the input simulation name is expected to be prepin.inp with the output files appended with <dat>.
  • runbatch <batch file>: Runs a series of simulations in sequence in the order defined in the text file <batch file>. The absolute or relative (to the directory from which the script is launched) path to each simulation file is defined on a separate line as shown in the example below. After completion, the script returns to the directory from which it was launched.

An example of a <batch file> for a Linux system is given below (for Windows forward slashes must be replaced with the backward ones). Note that the paths to the last three simulations in this example are given relative to the directory where the runbatch script is executed.

/home/flow3d/examples/weir/prepin.<ext1> <ext2>
/home/joe/tesprob/prepin.inp
../test1/prepin.<ext1>
../test2/prepin.inp
local_directory/prepin.inp <ext2>

On Linux systems, the runscripts may also be used to submit batch runs from the problem directory using the at or batch system utilities.

When the runpre and runpost scripts are run, the new files created will overwrite the old files. However, runhyd and runall scripts will not run if the results data file flsgrf.<ext2> exists in the directory. The user must delete flsgrf.<ext2> before runall and runhyd can be run (this is to help avoid accidental overwriting of results data files). The old files can be deleted by typing: $ rm flsgrf.<ext2> on Linux systems and $ del flsgrf.<ext2> on Windows.

Using the Runscripts¶

The runscripts are designed to be executed from the working problem directory. The scripts reside in the $F3D_HOME/local directory. In order to use the scripts the following environment variables must be set:

  • F3D_HOME: Specifies the path to the FLOW-3D installation directory.
  • F3DTKNUX_LICENSE_FILE: Specifies the location of the FLOW-3D license server.
  • PATH: The PATH environment variable must be modified to include $F3D_HOME/local otherwise the runscripts will not be found.
  • F3D_VERSION: Specifies which solver version to use. Valid options are double for the double precision version and prehyd for the custom double precision solver.

To run from the command line:

  1. Open a command prompt or terminal.
  2. Set the necessary environment variables.
    • Windows: This can be done by copying the environment from the batch file used to launch FLOW-3D . The contents of the batch file can be accessed by right-clicking on the FLOW-3D icon and choosing Edit.
    • Linux: Source the file flow3dvars.sh from the local directory of the installation.
  3. Change to the directory where the solver is running.
  4. Enter the desired runscript command, e.g. runhyd <ext2>

Using the Monitoring Program (PEEK)¶

PEEK is a menu-driven program that gives users basic control over the FLOW-3D solver while it is running. PEEK may be used when the solver is running either interactively or in batch mode. To run PEEK, just enter peek in a command prompt or terminal in the directory where the problem is running.

PEEK is most often used to check the status of a solver run by displaying the contents of the solver message file hd3msg.*. If the progress of the analysis is slower than anticipated and status messages are not printed out often enough to know how the process is running, the user may want to force a status message or tell the solver to write spatial or history data dumps to flsgrf. This is particularly useful when the analysis appears to be having trouble and the user would like to see what is going on at the current stage of the calculation. The simulation can also be terminated if the analysis is not running as expected. This option forces the current solution state to be written out, all data buffers to be dumped and all files to be closed before the solver is terminated. If the solver is terminated by a system kill, i.e., <Ctrl>c, data may be lost and the user will not be able to determine the solution state before termination.

PEEK communicates with the solver through a text signal file named HYdr3D_SIgnAL.dat. Each time PEEK is invoked, this file is created and a number indicating the desired action is written to the file. Once the solver reads the signal file, it is deleted. During execution, the solver checks signal flags (sent by PEEK) at the end of each time step. PEEK can instruct the solver to do a variety of things. The options most commonly used are shown below:

OPTION SIGNAL DESCRIPTION
0   Exit the peek utility
1 2 Terminate the solver after writing the current state to the output files
2 3 Write a status message to the hd3msg.* file
3 4 Write a restart data point to the flsgrf.* file
4 5 Write a history data dump to the flsgrf.* file
5 6 Pause the simulation
6 7 Resume the simulation
7   Display the entire hd3msg.* file. PEEK assumes that the message file is hd3msg.dat unless otherwise specified.
8   Display the end of the hd3msg.* file. PEEK assumes that the message file is hd3msg.dat unless otherwise specified.
9   Show the available restart times listed in the hd3msg.* file. PEEK assumes that the message file is hd3msg.dat unless otherwise specified.
10   Specify the name of the message file. PEEK assumes that the message file is hd3msg.dat unless otherwise specified.
